这是一个规范对话内容到项目文档的组织流程，可以确保文档清晰、准确、易懂。

> **注意**：只有当用户明确要求时（例如“请把这个总结成文档”），才使用此 skill。否则，继续正常对话。

## 使用场景

- 总结对话内容为文档
- 整理 vitest 学习经验
- 零散知识点归档

## 文档命名与基础规则

### 命名规范

**格式：** `NNN-xxx.md`

- `NNN` = 从 001 开始的递增编号
- `xxx` = 小写英文主题，连字符分隔

**示例：** `001-expect-custom-message.md`

### 文档内容质量标准

| 维度     | 要求                                 |
| -------- | ------------------------------------ |
| 清晰性   | 表述清晰，逻辑通顺，适合初学者理解   |
| 准确性   | 代码示例正确，API 调用符合官方文档   |
| 完整性   | 知识点完整，无遗漏重要使用场景       |
| 易理解性 | 初学者能理解，概念解释充分，流程清晰 |

### 文档质量审查流程

- **subagent 编排**：由 dispatcher 负责任务编排，reviewer 负责审查，modifier 负责修改
- **单阶段多轮审校**：第 1 轮进行全量初审；若存在问题，则修改后进入复审；后续轮次根据上一轮暴露的风险点做更深入的分析
- **问题驱动**：是否进入下一轮，不由固定轮次决定，而由阻塞问题是否清零、文档是否达到发布标准决定
- **用户介入条件**：默认由 agent 自动推进；若出现需求不清、来源冲突、无法验证等情况，dispatcher 应回退给用户确认

## 自动化审查-修改循环

### 循环架构

```
用户
  ↓
Dispatcher (流程管理器)
  ├─→ Reviewer (每轮评审)
  │    执行全量审查 → 标记阻塞问题/非阻塞问题/风险点
  │
  ├─→ 决策判断
  │    达标 → 发布检查
  │    有问题 → 派发给 Modifier 修改文档
  │    需求不清/来源冲突/无法验证 → 返回用户确认
  │
  └─→ Modifier (按需修改)
       根据问题单修改文档 → 返回下一轮评审
```

### Reviewer 工作流

**第 1 轮（全量初审）**

对文档进行完整审查，覆盖以下方面：

```
输入：初始文档
↓
审查维度：
  1. 清晰性 - 表述是否清楚，逻辑是否顺畅
  2. 准确性 - API、配置、命令、示例是否与可信来源一致
  3. 完整性 - 是否覆盖当前主题的必要知识点与常见场景
  4. 初学者适配度 - 是否适合有前端经验但缺乏 Vitest 深入经验的读者
  5. 来源与验证 - 是否标注来源，代码示例是否已验证或明确标注待验证
↓
生成评审报告：
  - 阻塞问题：不解决则不能发布
  - 非阻塞问题：建议优化，但不一定阻塞发布
  - 风险点：需要在后续轮次重点深挖的部分
  - 具体问题、位置、改进建议
  - 综合判断：直接通过 / 修改后复审 / 需要用户确认
↓
输出：完整评审报告
```

**第 2+ 轮（复审阶段）**

- 检查上一轮问题是否被妥善处理
- 检查修订是否引入新问题或副作用
- 针对上一轮暴露出的高风险区域做更深入分析
- 重新判断文档整体是否已达到发布标准
- 仍需覆盖全部维度，但审查精力应根据当前风险分布动态调整，不要求平均用力

### Modifier 工作流

接收评审报告后按以下原则执行修改：

```
输入：评审报告 + 文档版本 + 历史审校记录
↓
1. 优先修复阻塞问题，再处理非阻塞问题
2. 修改应严格围绕 reviewer 的问题单进行，不擅自扩大主题范围
3. 若问题涉及事实、来源或示例验证，必须补充证据或显式标注无法验证
4. 每次修改后记录修改说明：
   - 已处理的问题
   - 未处理的问题及原因
   - 需要 reviewer 在下一轮重点检查的风险点

↓
输出：修改后的文档 + 修改清单
```

### Dispatcher 决策流程

每轮评审结束后，Dispatcher 按以下规则决策：

| 条件                                     | 行动                             |
| ---------------------------------------- | -------------------------------- |
| 无阻塞问题，且满足发布条件               | 进入发布检查                     |
| 存在阻塞问题                             | 派发给 modifier 修改后复审       |
| 无阻塞问题，但 reviewer 标记存在高风险点 | 进入下一轮深入复审               |
| 需求不清、来源冲突、无法验证             | 中止自动流程，返回用户确认       |
| 连续多轮无实质进展                       | 中止并输出当前问题清单与阻塞原因 |

### 发布条件

满足以下**所有**条件才能发布：

1. ✓ 无阻塞问题遗留
2. ✓ 内容适合 Vitest 初学者理解，符合 AGENTS.md 中的目标读者要求
3. ✓ 技术事实已通过可信来源验证，或明确标注未验证部分
4. ✓ 代码示例与命令已验证可用，或明确标注验证状态
5. ✓ 信息来源已标注，优先使用官方文档，其次才是项目内资料或经验总结
6. ✓ 文件名符合格式：`NNN-xxx.md`
7. ✓ 发布前扫描 `docs/` 现有编号，分配下一个可用编号
8. ✓ Markdown 结构、链接、代码块可正常使用

任何一项不满足 → 拒绝发布，返回问题清单
