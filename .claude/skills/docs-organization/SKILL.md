规范对话内容到项目文档的组织流程，确保文档清晰、准确、易懂。

> **注意**：只有当用户明确要求时（例如“请把这个总结成文档”），才使用此技能。否则，继续正常对话。

## 使用场景

- 总结对话内容为文档
- 整理 vitest 学习经验
- 零散知识点归档

## 核心规则

### 1. 命名和编号

**格式：** `NNN-xxx.md`

- `NNN` = 从 001 开始的递增编号
- `xxx` = 小写英文主题，连字符分隔

**示例：**

- `001-expect-custom-message.md`
- `002-test-setup-teardown.md`

**❌ 红旗：** 无编号、中文主题、大写字母、不连续编号、空格或特殊字符

**验证命令：** `ls -1 docs/*.md | sort`

### 2. 内容质量

| 维度     | 要求                                        |
| -------- | ------------------------------------------- |
| 可理解性 | 适合初学者，避免过度术语，提供代码示例      |
| 准确性   | 信息可追溯，内容已验证，代码可运行          |
| 来源标注 | 注明来源（文档/测试/搜索），附 URL 或版本号 |

**来源示例：**

```markdown
## 来源

- [vitest 官方文档 - expect API](https://cn.vitest.dev/api/expect)
- 个人测试验证（2025-04-05）
```

### 3. 审查流程（必须）

使用 subagent 进行多轮审查：

1. **语言清晰性** - 表述清晰、逻辑通顺
2. **技术准确性** - 验证代码和概念
3. **完整性** - 检查遗漏
4. **易理解性** - 确认初学者能理解

**❌ 审查不完整：** 未经审查直接提交、仅一轮审查、代码未验证

**✅ 审查完成：** 多轮审查改进、来源已标、代码已验证

### 4. 文档位置

全部放入 `docs/` 目录：

```
docs/
  001-xxx.md
  002-yyy.md
```

## 常见错误

| 错误       | 示例                                 | 正确                               |
| ---------- | ------------------------------------ | ---------------------------------- |
| 跳过审查   | 直接提交                             | subagent 多轮审查                  |
| 命名不规范 | `vitest-expect.md` / `001_Expect.md` | `001-expect-api.md`                |
| 来源缺失   | "vitest 支持..."                     | "vitest 支持...（来源：官方文档）" |

## 工作流程

当用户确认需要保存内容到文档时：

1. **总结对话内容**
2. **查看现有文档** → `ls -1 docs/*.md` 确定编号
3. **编写初稿**
4. **subagent 审查** → 多轮改进
5. **保存** → `docs/NNN-xxx.md`

## 质量检查清单

- [ ] 文件名格式：`NNN-xxx.md`
- [ ] 编号连续，位置正确
- [ ] 内容适合初学者
- [ ] 来源已标注
- [ ] 代码可运行
- [ ] 经 subagent 多轮审查
- [ ] 无语法错误，结构清晰
