本技能规范了如何将对话内容组织和总结成项目文档。遵循统一的文档命名、编号和审查流程，确保文档质量清晰、准确、真实有效、易懂。

## 什么时候使用

- 用户要求将对话内容总结成文档
- 需要为学习 vitest 的经验归档成文档
- 整理零散知识点成项目文档
- 文档需要放在 `docs/` 目录下

## 核心规则

### 1. 文档命名和编号

**格式：** `NNN-xxx.md`

其中：

- `NNN` = 文档编号（从 001 开始，按创建顺序递增）
- `xxx` = 文档主题（英文、小写、用连字符分隔单词）

**示例：**

- `001-expect-custom-message.md` - 关于 vitest expect 的自定义错误消息
- `002-test-setup-teardown.md` - 关于测试的设置和清理
- `003-async-test-patterns.md` - 异步测试模式

**红旗警告：**

- ❌ 文档名称不包含数字编号
- ❌ 文档主题使用中文或大写字母
- ❌ 编号不连续（缺少中间编号）
- ❌ 文档名称包含空格或特殊字符
- ❌ 文件名不以 `.md` 结尾

**验证步骤：**
运行以下命令检查现有编号，确保新文档编号正确：

```bash
ls -1 docs/*.md | sort
```

### 2. 文档内容质量要求

文档必须满足以下条件：

**可理解性：**

- [ ] 内容适合初学者理解
- [ ] 使用清晰的语言，避免过度技术术语
- [ ] 提供实际示例（代码片段或使用场景）
- [ ] 可选：添加对测试在实际开发中的应用场景的解释

**准确性：**

- [ ] 信息源可追溯（标注来源）
- [ ] 内容经过验证（不是猜测）
- [ ] 代码示例可运行
- [ ] 没有误导性的声明

**来源标注：**

- [ ] 必须注明信息来源（官方文档、个人测试、web 搜索等）
- [ ] 如来自网络，需附上 URL 链接
- [ ] 如来自官方文档，标注版本号

**示例：**

```markdown
## 来源

- [vitest 官方文档 - expect API](https://cn.vitest.dev/api/expect)
- 个人测试验证（2025-04-05）
```

### 3. 审查流程

文档编写完成后，**必须**采用 subagent 进行内容自我审查修改流程，流程包括：

**审查步骤：**

1. **语言清晰性审查** - 确保表述清晰、逻辑通顺
2. **技术准确性审查** - 验证代码示例和概念准确
3. **完整性审查** - 检查是否遗漏重要信息
4. **易理解性审查** - 确认初学者能理解

**审查目标：**

- 内容质量更加清晰、准确
- 信息真实有效
- 易于理解和实际应用

**红旗警告 - 审查不完整的标志：**

- ❌ 文档未经审查直接提交
- ❌ 只做了一轮审查就停止
- ❌ 审查后仍有不清晰或歧义的地方
- ❌ 代码示例未验证可运行

**审查完成的标志：**

- ✅ 经过 subagent 多轮审查并改进
- ✅ 内容清晰、准确、真实有效、易懂
- ✅ 所有来源已标注
- ✅ 代码示例已验证可运行

### 4. 文档放置位置

所有文档必须放在 `docs/` 目录下：

```
docs/
  001-xxx.md
  002-yyy.md
  003-zzz.md
  ...
```

## 常见错误

### ❌ 错误做法 1：跳过审查流程

```
文档完成后直接提交，未经 subagent 审查
```

**为什么有问题：**

- 文档可能有不清晰或歧义的地方
- 技术信息可能不准确
- 初学者可能不理解

**正确做法：**

- 文档完成后，使用 subagent 进行多轮审查
- 持续改进直到达到高质量标准
- 明确标注所有来源

### ❌ 错误做法 2：不规范的命名

```
vitest-expect.md  # 缺少编号
001_expect.md     # 使用下划线而非连字符
001-Expect-API.md # 使用大写字母
```

**正确做法：**

```
001-expect-api.md  # 编号-小写主题
```

### ❌ 错误做法 3：来源信息不足

```
"vitest 支持自定义 expect 消息"
```

**正确做法：**

```
"vitest 支持自定义 expect 消息（来源：vitest 官方文档 - expect API）"
```

## 完整工作流程

1. **对话和总结** - 与用户讨论 vitest 知识点，准备总结
2. **查看现有文档** - 运行 `ls -1 docs/*.md` 确定下一个编号
3. **编写初稿** - 撰写文档初稿
4. **启动审查流程** - 使用 subagent 进行多轮审查修改
5. **审查完成** - 确认文档已达到高质量标准
6. **提交文档** - 将文档保存到 `docs/NNN-xxx.md`
7. **验证** - 检查文档在正确位置且格式正确

## 项目应用示例

**场景：** 用户学习了 vitest 的 expect 自定义消息功能

**流程：**

```
1. 对话总结内容
2. 确认下一个编号：运行 ls -1 docs/*.md
   结果：001-xxx.md 已存在，下一个是 002
3. 编写文档：002-expect-custom-message.md
4. 启动 subagent 审查
   - 审查 1：内容清晰性 → 修改
   - 审查 2：技术准确性 → 验证代码
   - 审查 3：初学者理解度 → 简化表述
5. 审查完成，达到高质量标准
6. 保存文档到 docs/002-expect-custom-message.md
```

## 质量检查清单

在文档提交前，检查以下项目：

- [ ] 文件名格式：`NNN-xxx.md`（N 为数字，xxx 为小写主题）
- [ ] 编号连续且正确
- [ ] 文件位置：`docs/` 目录
- [ ] 内容适合初学者理解
- [ ] 所有信息源已标注
- [ ] 代码示例可运行
- [ ] 经过 subagent 多轮审查
- [ ] 内容质量达到：清晰、准确、真实有效、易懂
- [ ] 无语法或拼写错误
- [ ] 文档结构清晰（有标题、小节等）
