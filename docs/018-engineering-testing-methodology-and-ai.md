# 工程化测试方法论：从「为什么测」到「怎么落地」（含 AI 辅助）

> 本文是一套**把 Vitest 用进真实项目的决策框架**，不是 API 教程。它回答的是工程化问题：为什么投入测试、看哪些指标、测什么、怎么写、怎么让它长期活着，以及 **AI 能写测试后这套方法论怎么变**。
> 具体 API（`expect` / `vi.mock` / 覆盖率配置等）请看对应主题页与 `docs/` 其它文档。

## 先看结论

测试的工程化决策有**层级**，上层决定下层，不能从中间切入：

| 层                  | 回答的问题                         | 谁主导              | 本文默认结论                                                                     |
| ------------------- | ---------------------------------- | ------------------- | -------------------------------------------------------------------------------- |
| **L0 价值层**       | 为什么测？看哪个指标？投到哪收手？ | 人（AI 碰不到）     | 主轴=变更信心/防回归；北极星=漏到线上的 bug 数；护栏=flaky≈0；投入按边际收益递减 |
| **L1 策略层**       | 测什么？测试怎么分布？             | 人（AI 碰不到）     | 边界=对外公共 API；形状=前端奖杯·后端金字塔；选择=风险驱动+不测清单              |
| **L2 战术层**       | 具体怎么写每个用例？               | AI 主力 + 人审      | mock 只在 IO 边界；用例=等价类+边界值+错误路径；覆盖率只找漏不当 KPI             |
| **L3 运营层**       | 怎么不让套件越跑越慢？             | 工具+CI             | 分层反馈：内循环只跑快 unit，慢测进 CI                                           |
| **★ AI 层（横切）** | AI 写测试，方法论怎么变？          | 人定规则 + 自动护栏 | 实现=上下文、规格=断言源；自动化护栏为主、人只审意图                             |

> 上表里的「北极星 / 护栏 / 奖杯 / 金字塔 / flaky / 棘轮」等名词，下文各小节都有白话解释，第一次读到表格看不懂是正常的，可先扫一眼往下读。

**一句话主线**：写测试在 AI 时代变便宜了，于是瓶颈从「写」转移到「判断该测什么（L0/L1）」和「验证测得对不对」——而这两头恰恰是 AI 干不了、必须人来的。所以 **AI 越能写测试，L0 和验证门禁的权重越高，而不是越低**。

## L0 价值层：先想清楚为什么投入

工程化的本质是「在约束下做权衡」。脱离约束谈「好测试」，只会得到「要反映业务、要考虑边界、要易重构」这类正确但无法指导取舍的口号。所以第一步不是「怎么测」，而是**测试到底解决什么问题、怎么证明它值**。

### 价值主轴：变更信心 / 防回归

测试的价值有多种主张（防回归、缺陷左移、设计驱动、替代手动测），但方法论必须选一个**主轴**，因为它决定你该盯哪些指标。本文选 **「变更信心 / 防回归」**：测试的首要价值是让系统**敢改、敢重构、敢发**，把回归风险从人脑转移到自动化。

收益链由主轴推导，不需要单独罗列：

> 敢重构敢改老代码 → 回归 bug 减少 → 线上事故/救火减少 → 替代部分手动回归省工时 → 新人敢动代码（测试即安全网+文档）→ 迭代提速

这条链上每一环都能回归到北极星指标。

### 指标：一个北极星 + 一个护栏

指标要分清**滞后指标**（反映结果，真实但慢，事后才知道）和**领先指标**（反映过程，快但易被无关因素污染），并且只设一个北极星，避免仪表盘堆砌。

- **北极星 = 漏到线上的 bug 数（逃逸缺陷）**：测试没拦住、跑到生产才被发现的 bug 数。最直观、最贴业务痛，越少说明网越密。
- **护栏 = flaky 率 ≈ 0**：同一份代码测试一会过一会不过。这不是收益指标而是护栏——只要测试时不时误报红灯，开发者就会开始无视它，「变更信心」瞬间归零。
- 进阶可补 **变更失败率**（发布的变更里多少比例上线出事，需回滚/热修）——它是 DORA 研发效能四指标之一。DORA 是业界衡量交付效能的一套标准指标，四项为：部署频率（Deployment Frequency）、变更前置时间（Lead Time，从提交到上线多久）、变更失败率（Change Failure Rate）、故障恢复时间（MTTR，出事后多久修好）。
- **不要把覆盖率、测试数量当北极星**：它们是产出（output）不是结果（outcome），容易被刷。

### ROI 边界：边际收益递减 + 阶段感知

测试不是免费的，有「写 + 跑 + 维护」三份成本。工程成熟度的标志不是测得多，而是**知道何时收手**：

- 预算先砸在**高风险核心路径**（影响 × 出错概率 × 变更频率最高处），ROI 最高。
- 当「再加测试拦下的 bug 成本 < 写+维护这测试的成本」时就停手。
- **阶段感知**：探索期 / 原型 / 需求剧变期主动少测甚至不测——此时需求本身不稳定，测试会被频繁推翻，ROI 为负。

## L1 策略层：测什么、怎么分布

### 被测单元边界 = 对外公共 API

从模块/组件的**对外接口**调用，内部协作者一律走真实实现。这样重构内部实现时测试不动，是「易重构」的结构性保证。反例是「逐函数隔离测、把所有协作者都换成桩」——粒度细、定位快，但与内部结构强耦合，一重构就崩。

### 测试形状 = 前端奖杯 / 后端金字塔

「形状」指在范围轴（unit 单元 / integration 集成 / e2e 端到端）上各写多少的比例——画出来像什么形状，就叫什么。本文按两边生态分开：

- **前端用测试奖杯（Testing Trophy）**：中间的集成层最胖，即**多写集成测试**，重心压在「渲染后从用户视角测」，理由是该层性价比最高。
- **后端用测试金字塔**：底座最宽，即**大量纯逻辑 unit 打底** + 往上一层走真实 DB 的集成，越往上越少。

> 分层的本质不是「代码量大小」，而是「隔离程度——让多少真实代码一起跑」。

### 测什么 = 风险驱动 + 显式不测清单

不是「能测都测」（最贵最慢最脆），也不是「覆盖率驱动」（奖励刷无意义测试）。而是：

- 按「业务影响 × 出错概率 × 变更频率」排优先级，核心链路 / 复杂分支 / 出过 bug 处重测。
- **显式列出不测清单**：琐碎 getter、框架胶水、纯配置、一次性脚本。

## L2 战术层：每个用例具体怎么写

### mock 线 = 只在进程/IO 边界

只 mock 你**无法控制 / 不确定 / 慢**的东西：网络请求、当前时间、随机数、真正的第三方 SaaS。**凡是自己写的代码（含 DB 访问层、内部 service）一律走真实**。

> 现实里绝大多数脆测试，是因为把「自己写的东西」也 mock 了（最典型：测 service 时把自己的 repository / 另一个内部 service 换成假数据）。一旦 mock 了自己的代码，重构内部协作时测试照样崩，「抗重构」就没了。

后端 DB 用内存库 / 测试容器代替生产库，但**保留真实 DB 行为**，不要换成假数据桩。

### 用例设计 = 等价类 + 边界值 + 错误路径

「考虑边界」如果靠脑补穷举，必然漏。要用**系统化、可复现**的方法从一个被测目标推导出该写哪些用例：

- **等价类划分**：把输入分成若干等价类，每类挑一个代表，避免冗余。
- **边界值分析**：空 / 0 / 最大 / null / 越界 / off-by-one。
- **错误与异常路径必覆盖**（不只测理想路径）。
- 复杂纯逻辑可上**基于性质的测试（property-based，如 fast-check）**：不逐个举例子，而是断言一条"无论输入什么都应成立的规律（不变量）"，由工具随机生成大量输入去试图推翻它。例：无论数组是什么，`reverse(reverse(arr))` 都应等于 `arr`。

Vitest 里组织多组输入用 `test.each` 最省事（官方推荐）。

### 覆盖率 = 只找漏，不当 KPI

覆盖率能 100% 而测试啥都没抓到（断言写空、或只调用不断言照样算覆盖）。所以**只看数字会优化出一堆假测试**。本文用法：

- **CI 不卡「达到 80%」这种绝对值**，改成卡「**不允许下降** + 变更行必须被覆盖」。
- Vitest 的 `coverage.thresholds.autoUpdate` 正是「不下降」机制——当前覆盖更高时，自动把阈值上调并写回配置文件，从而锁死、禁止后续下滑（即「棘轮效应 ratchet」：只升不降）。
- 用 **glob 键**给关键模块单独设阈值，用 `perFile` 按文件卡，避免全局平均数掩盖核心区漏测。
- **看分支覆盖（branches）优先于行覆盖**。Vitest 支持 `lines / functions / branches / statements` 四种指标。
- provider 默认 `v8`（详见 `docs/017`），跨非 V8 运行时才切 `istanbul`。
- **真正证明测试有效的是变异测试**：注入一个 bug，看测试抓不抓得到。它不在乎测试是谁写的，是「假测试」的客观裁判。Vitest 可用 StrykerJS（`@stryker-mutator/vitest-runner`）跑，对关键模块定期做。

## L3 运营层：让套件长期活着

测试套件越大越慢，慢到一定程度开发者就不跑了，测试就废了。所以**反馈速度是测试能否存活的关键工程杠杆**。默认策略是**分层反馈**：

- **本地内循环**：只跑快 unit，配合 watch 模式 + 只跑变更相关，追求秒级反馈。
  - `vitest --changed [commit]`：只跑与 git 变更（暂存+未暂存，或指定提交/分支）相关的测试。
  - `vitest related <files> --run`：只跑覆盖指定源文件的测试（适合 lint-staged）。
- **CI 门禁**：跑全量（含慢的集成 / 真 DB 测试），用分片并行 + 按耗时找慢测优化。
  - `vitest run --shard=<index>/<count>`：CI 多机分片并行（只能在 run 模式用，不能配 `--watch`）。
  - `pool` 选项控制并行方式：`'threads' | 'forks' | 'vmThreads' | 'vmForks'`，**默认 `'forks'`**。
- **CI 务必用 `vitest run`（或 `--run`）** 确保不进入 watch、进程能正常退出。

## ★ AI 层：AI 能写测试后，方法论怎么变

> 本节结论已用 Vitest 官方文章《Writing Tests with AI》校验。

**核心反转**：AI 没让方法论失效，而是让它**换了形态和载体**。AI 改变的只有 L2（把用例翻译成测试代码，这本就是机械活、是 AI 主场），而它**碰不到甚至会主动破坏 L0/L1**——AI 不知道你的业务里哪条路径最关键、哪块在剧变期不值得测，它会对所有东西一视同仁地生成测试。于是：

> 人的角色：**写测试的人 → 定规格 + 审意图的人**。
> 方法论的形态：**一篇给人看的指南 → 一份约束 AI 的 `AGENTS.md` 规则 + 一套自动化评分卡**。

### 事实源 = 角色分离（实现当上下文，规格当断言源）

官方明确建议**要给 AI 看实现**（完整源文件、类型、现有测试、配置），否则 AI 写出的测试不具体、API 用错、不合项目约定。但官方同时警告 AI 倾向**过度实现耦合**（过度 mock、断言内部调用顺序，一重构就崩）。两段合起来 = **角色分离**：

- **实现代码 → 给 AI 当「写法上下文」**：拿到真实类型、签名、约定、具体值，保证写得对。
- **规格 / 意图 → 当「断言的事实源」**：决定对错标准，保证测的是行为不是结构。
- 显式约束 AI：断言基于行为，不得断言内部实现细节。

这样既符合官方「要看实现」，又守住了「抗重构」。**绝不能让「看了实现」污染到「断言依据」**——否则 AI 把当前实现（含 bug）快照成「正确行为」，测试与实现同源，重构必崩、bug 被锁死。

### 验证门禁 = 自动化护栏为主，人只审意图

AI 能量产测试 → **人工逐条 review 变成新瓶颈**（人审到第 50 个必然橡皮图章）。如果验证靠人审，只是把成本从「写」挪到「审」。所以要**把评分卡尽量自动化**：

- **lint / 配置拦机械陷阱**：
  - 禁 `jest.fn` / `jest.mock`（AI 训练数据偏 Jest，常误用，应是 `vi.fn` / `vi.mock`）。
  - 禁只有 `expect(x).toBeDefined()` 这类弱断言。
  - 限制 / 禁用快照（snapshot 是「假测试」重灾区）。
  - 配置 `restoreMocks: true` 防 mock 跨测试泄漏（AI 常忘恢复）。
- **变异测试当 AI 测试有效性的客观裁判**：直接验「这测试到底抓不抓 bug」，中和 AI 最大的弱断言问题。
- **人只审机器判不了的那一件事——断言是否对齐业务意图**。
- 官方还建议：用 `AGENTS.md` 承载项目约定（AI 自动识别遵循）；生成后立即 `vitest run` 验证；用自然语言迭代（「这些 mock 太多，改成测真实模块的集成」比逐条细抠更有效）。

### 收口的几条约定（低层级，直接并入 AGENTS.md）

- **TDD**：不强制（本文主轴是变更信心而非设计驱动）；复杂纯逻辑可选用先写测试。
- **测试数据**：工厂 / builder 优先于内联字面量与共享 `beforeEach`；Vitest 可用 `test.extend` 做 fixture。
- **快照**：默认禁用；仅 `toMatchInlineSnapshot` 对极小、稳定结构例外。

## 常见误区

- **「易重构」和「靠覆盖率优化」当成可以同时无脑追求**：覆盖率天然把你往「贴实现写」拽，与抗重构有张力；解法是 mock 线划在 IO 边界 + 覆盖率只找漏不当 KPI。
- **把覆盖率 / 测试数量当北极星**：它们是产出不是结果，会被刷出虚高，逃逸 bug 数却不动。
- **测自己的代码时把内部协作者也 mock 了**：这是脆测试头号来源，直接毁掉抗重构。
- **从中间层级（L2 怎么写）切入做方法论**：不先定 L0 价值主轴，就无法回答「看哪些指标 / 投到哪收手」。
- **以为 AI 能写测试 = 方法论可以省掉**：恰恰相反，判断该测什么和验证测得对不对的权重反而上升。
- **让 AI「看着代码写测试」且不约束断言依据**：会产生实现耦合 + 弱断言的同源测试，覆盖率好看但抓不到 bug。
- **AI 测试只靠人审**：量产后人是瓶颈，必须自动化护栏（lint + 配置 + 变异测试）兜底。
